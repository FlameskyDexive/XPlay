# 11. 商业化与公平性设计

## 1. 模块目标

- 建立“效率付费、非强制付费”的轻度修仙商业化体系。
- 坚守主线公平性，不以付费或SLG活跃绑核心成长。
- 支撑长期运营收益与玩家信任并存。

## 2. 商业化点位设计

## 2.1 月卡/战令

- 提供离线上限、便捷功能、外观权益。
- 不提供突破唯一材料或绝对战力门槛道具。

## 2.2 外观付费

- 角色皮肤、法宝特效、洞府装饰。
- 与战力弱绑定或不绑定，强调展示价值。

## 2.3 资源付费

- 售卖非唯一可替代资源包。
- 采用限次与分段投放，防止数值崩盘。

## 2.4 纯修仙付费闭环

- 月卡、战令、便捷功能应优先服务修炼、洞府、PVE和构筑调整。
- 不以宗门职位、SLG参与次数或组织排名作为高价值付费权益前置条件。
- 纯修仙用户也应存在稳定付费点：外观、挂机便捷、养成容错、收集扩展。

## 3. 公平性硬约束

- 不售卖“突破必需唯一材料”。
- 不以SLG参与决定主线强度上限。
- 付费提升效率，不直接越级碾压。
- 不允许通过“宗门专属礼包”间接制造不参加SLG就吃亏的感受。

## 4. 付费与玩法联动设计

- 将便捷能力聚焦于“省时间”，而非“跳机制”。
- 付费道具对雷劫应体现“容错提升”，不是“无脑通关”。
- 战令奖励需通过边界标签系统过滤越界资源。

## 5. 代码设计要点（ET）

## 5.1 建议包归属

- `cn.etetet.gacha`：抽卡与保底。
- `cn.etetet.item`：商城道具与发放。
- `cn.etetet.cultivation`：效率加成生效点。

## 5.2 建议组件

```csharp
public class CommerceBenefitComponent : Entity, IAwake
{
    public int MonthlyCardLevel;
    public int BattlePassLevel;
    public long BenefitExpireTime;
}
```

## 5.3 建议系统

- `CommerceGrantSystem`：权益发放与有效期管理。
- `RewardBoundaryCheckSystem`：奖励标签过滤与越界拦截。
- `PayValueMonitorSystem`：付费效率收益监控与告警。

## 6. 配置与协议

## 6.1 配置表建议

- `ShopPackConfig`
- `BattlePassConfig`
- `RewardBoundaryConfig`

## 6.2 协议建议

- `C2M_ShopBuy` / `M2C_ShopBuyResult`
- `C2M_BattlePassClaim` / `M2C_BattlePassSync`

## 7. 运营监控指标

- 付费用户与非付费用户主线推进差距。
- 单位付费效率收益比与版本波动。
- 因“Pay to Win”反馈导致的负面舆情占比。
- 纯修仙付费用户的30日留存与付费转化稳定性。
