# 14. 修仙协议迁移说明

## 1. 目标

将修仙主循环协议从旧命名迁移到新命名：

- 旧：`C2M_Cultivate` / `M2C_CultivateResult`
- 新：`C2M_CultivateStart` / `M2C_CultivateSync`

- 旧：`C2M_BreakRealm` / `M2C_BreakRealmResult`
- 新：`C2M_RealmBreakTry` / `M2C_RealmBreakResult`

迁移策略：**兼容并行，逐步收口**。

## 2. 当前状态

- 新协议已完成定义、代码生成与处理器落地。
- 旧协议处理器仍保留，且已增加废弃日志与 `[Obsolete]` 标注。
- 客户端新增统一调用入口 `CultivationHelper`，后续新功能必须走新协议。

## 3. 协议与Opcode

保持旧号不变，新增号在末尾追加：

- 旧协议：
  - `C2M_Cultivate = 10301`
  - `M2C_CultivateResult = 10302`
  - `C2M_BreakRealm = 10303`
  - `M2C_BreakRealmResult = 10304`

- 新协议：
  - `C2M_CultivateStart = 10314`
  - `M2C_CultivateSync = 10315`
  - `C2M_RealmBreakTry = 10316`
  - `M2C_RealmBreakResult = 10317`

## 4. 推荐调用方式（客户端）

```csharp
// 开始修炼（新协议）
M2C_CultivateSync cultivateSync = await CultivationHelper.StartCultivate(scene, offlineQueueHours: 12);

// 尝试突破（新协议）
M2C_RealmBreakResult breakResult = await CultivationHelper.TryRealmBreak(scene, autoMode: false);
```

## 5. 兼容约束

- 不删除旧协议消息定义与旧处理器，直到所有调用点切换完成。
- 新增任何修仙调用代码时，禁止继续使用旧协议。
- 旧处理器日志关键字：`CultivationProtocolDeprecated`，可用于日志平台统计。

## 6. 旧协议禁用开关（灰度）

可通过环境变量控制是否拒绝旧协议请求：

- 变量名：`ET_CULTIVATION_DISABLE_LEGACY_PROTOCOL`
- 关闭（默认）：不设置该变量，或设置为 `0/false/off/no`
- 开启禁用：设置为 `1/true/on/yes`

开启后，旧协议 `C2M_Cultivate` / `C2M_BreakRealm` 会直接返回错误：

- `ERR_LegacyProtocolDisabled`

推荐灰度方式：

1. 先在测试环境开启，确认客户端已全部走新协议。
2. 小流量区服开启并观察告警日志。
3. 全量开启后保留一版观察周期，再删除旧处理器。

## 7. 下线建议节奏

1. 第1阶段（当前）：新旧并行，统计旧协议调用量。
2. 第2阶段：旧协议调用量降到阈值后，客户端移除旧消息发送入口。
3. 第3阶段：服务端关闭旧处理器注册，保留协议定义一版过渡。
4. 第4阶段：最终移除旧协议定义与生成代码。
