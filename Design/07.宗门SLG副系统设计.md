# 07. 宗门SLG副系统设计

## 1. 模块目标

- 作为“效率放大器 + 社交荣誉场”，不干扰修仙主进度独立性。
- 提供低压力、低在线强绑定的协作体验。
- 保证所有核心突破材料在纯PVE路径可稳定获取。

## 2. 定位与边界

- 定位：副玩法，不是强制主线内容。
- 原则：不做重外交、重战时在线绑定。
- 红线：禁止SLG独占境界突破必需材料。
- 占比：以20-30分钟标准日常为基准，SLG强制操作时长应控制在 `0-6分钟`。
- 开放节奏：首周仅软开放，不作为新手主漏斗前置。

## 3. 玩法内容细化

## 3.1 宗门建设

- 建筑：主殿、藏经阁、灵田、矿场、防御阵。
- 产出：效率类增益、宗门贡献、展示型内容。
- 升级节奏：短建造时长 + 多次可见收益反馈。

## 3.2 轻领地战

- 低频开战，战前预约参与。
- 战斗过程支持托管，减少强在线要求。
- 结果以战报呈现，强调策略配置而非手操时长。

## 3.3 宗门协作Boss

- 异步集火机制，成员分时段参与。
- 按贡献区间结算，不要求全员同屏。
- 奖励偏效率与荣誉，不给主线唯一材料。

## 3.4 宗门任务

- 任务类型：捐献、护送、采集。
- 目标：提供轻社交协作与稳定活跃行为。
- 支持自动执行或批量提交，降低操作负担。

## 4. 奖励边界实现

## 4.1 奖励类型

- 修炼效率加速道具。
- 外观、称号、社交展示奖励。
- 非唯一可替代资源包。

## 4.2 强约束机制

- 使用 `SLGRewardLimitConfig` 标记可投放资源标签。
- 运营配置发布前执行自动校验，禁止越界投放。
- 线上巡检发现越界条目时自动告警。

## 4.3 非参与补位设计

- 不参与SLG的玩家由个人周常、个人试炼链、洞府巡修获得同层级基础成长反馈。
- 补位内容只覆盖主线推进与基础效率，不覆盖宗门荣誉、职位权益和组织展示奖励。
- 若版本运营活动包含宗门奖励，需同步配置纯PVE向的替代活动入口。

## 5. 负担控制策略

## 5.1 时间占比红线

- 周结算窗口每周不超过 `2` 次。
- 单次需玩家主动参与的SLG操作不应强制持续超过 `15分钟`。
- 非活动窗口期，宗门玩法以托管、派遣、领取和查看战报为主。

## 5.2 常规减负手段

- 战报与调度全自动托管。
- 新手免战期 + 低损撤退规则。
- 建筑升级节奏偏轻量，避免“上线收工焦虑”。

## 6. 代码设计要点（ET）

## 6.1 建议包归属

- `cn.etetet.sect`：宗门组织、任务、协作Boss、战报。
- `cn.etetet.slgmap`：轻领地地图与冲突结算。

## 6.2 建议组件

```csharp
public class SectMemberComponent : Entity, IAwake
{
    public long SectId;
    public int Position;
    public int Contribution;
    public int WeeklyActivity;
}
```

```csharp
public class SectBattleProxyComponent : Entity, IAwake
{
    public int AutoDeploySchemeId;
    public long LastBattleTime;
    public int LastBattleResult;
}
```

## 6.3 建议系统

- `SectBuildSystem`：建筑升级、增益生效、成员权限校验。
- `SectMissionSystem`：任务派发、进度同步、批量结算。
- `SectBossSystem`：异步伤害结算、贡献排名、奖励分配。
- `SLGRewardBoundarySystem`：奖励校验与越界拦截。

## 7. 配置与协议

## 7.1 配置表建议

- `SectBuildingConfig`
- `SectMissionConfig`
- `SectBossConfig`
- `SLGRewardLimitConfig`

## 7.2 协议建议

- `C2M_SectAction` / `M2C_SectStateSync`
- `C2M_SectMissionSubmit` / `M2C_SectMissionSettle`
- `M2C_SectBattleReport`

## 8. 验收指标

- 纯修仙玩家主线进度不受SLG参与影响。
- 宗门系统7日留存与轻活跃达成目标。
- 因SLG压力退坑反馈占比持续下降。
- SLG日均主动操作时长占比保持在目标红线内。
