# 03. 雷劫天劫系统

## 1. 模块目标

- 将每次小层突破都做成可感知挑战，不做纯进度条升级。
- 通过“雷劫 + 心魔”双轴设计，强调修仙主题与策略构筑。
- 保障失败可恢复，避免长期硬卡导致流失。

## 2. 玩法内容细化

## 2.1 触发规则

- 小层突破 `N重 -> N+1重`：必触发雷劫。
- 大境跨越 `9重 -> 下境1重`：必触发“天劫 + 心魔劫”双阶段。
- 支持手动渡劫与自动渡劫两种模式。

## 2.2 劫难三轴机制

- 伤害轴：雷击段数、倍率、穿透。
- 干扰轴：沉默、逆脉、心魔幻象。
- 环境轴：灵气紊乱、治疗衰减、护盾衰减。

## 2.3 模板节奏（1->9）

- 1->2：入门雷劫，强调识别机制。
- 3->6：开始考验回复、抗性、冷却管理。
- 7->9：高穿透+控制组合，要求完整构筑。
- 跨境：双阶段Boss战，兼顾硬数值与道心判定。

## 2.4 失败补偿与保底

- 失败不掉大境界，仅回退当前层进度。
- 返还80%突破资源。
- 发放“劫灰”用于减伤、保命、抗心魔道具兑换。
- 连续失败触发“天道垂怜”，最多降低50%雷劫强度。

## 2.5 渡劫策略位

- 法宝位：抗雷/反震/护命。
- 丹药位：瞬回/持续恢复/免控。
- 阵法位：减伤阵/聚灵阵/净心阵。
- 护道人位：提供一次外援技能。

## 3. 战斗与结算流程

1. 玩家发起突破请求。
2. 服务端按境界层级构建劫难实例。
3. 战斗中按波次刷机制与环境惩罚。
4. 结算成功推进层级；失败执行返还与保底累计。
5. 下发结果并刷新相关系统入口与指引。

## 4. 代码设计要点（ET）

## 4.1 包与职责

- `cn.etetet.tribulation`：劫难生成、波次控制、结算补偿。
- `cn.etetet.cultivation`：突破状态与境界推进。

## 4.2 建议组件

```csharp
public class TribulationComponent : Entity, IAwake
{
    public int RealmGroupId;
    public int FromStage;
    public int ToStage;
    public int WaveIndex;
    public int TotalWaves;
    public int FailStreak;
    public bool IsCrossRealm;
}
```

## 4.3 建议系统

- `TribulationEnterSystem`：实例创建、参数采样、资源预扣。
- `TribulationWaveSystem`：波次驱动、机制注入、环境Debuff。
- `TribulationSettleSystem`：胜负结算、奖励发放、层级推进。
- `TribulationCompensationSystem`：失败返还、劫灰发放、保底更新。

## 4.4 安全与一致性

- 所有突破结算只信服务端结果，客户端仅做表现。
- 关键流程加幂等键（玩家Id + RealmGroupId + ToStage）。
- `await` 后通过 `EntityRef<TribulationComponent>` 重取实体再访问。

## 5. 配置与协议

## 5.1 配置表

- `TribulationConfig`：波次、机制池、奖励、失败返还比例。
- `TribulationDebuffConfig`：干扰轴与环境轴参数。
- `TribulationCompensationConfig`：劫灰掉落与保底曲线。

## 5.2 协议建议

- `C2M_TribulationEnter`
- `M2C_TribulationStart`
- `M2C_TribulationWaveSync`
- `M2C_TribulationSettle`

## 6. 验收指标

- 7日内人均小层雷劫体验次数 >= 10。
- 渡劫失败后次日流失率持续可控。
- 连续失败用户中，48小时内突破成功占比达目标阈值。
