# 02. 境界体系与突破系统

## 1. 模块目标

- 将角色成长拆分为 `18个大境界 × 9重小层`，形成长期目标池。
- 保证每次小层提升都有明确反馈，不做“纯数值点点点”。
- 通过突破门槛控制内容解锁节奏，避免系统过早膨胀。

## 2. 玩法结构细化

## 2.1 境界分段

- 凡道（搬血 -> 尊者）：建立基础战斗与养成理解。
- 神道（神火 -> 至尊）：引入流派深度与风险收益玩法。
- 仙道（真仙 -> 仙帝）：提供终局构筑、长期追求与巅峰挑战。

## 2.2 小层成长节奏（1-9重）

- `1-3重`：快速进阶，强化新手爽感。
- `4-6重`：进入构筑门槛，开始需要法宝/丹药/阵法配合。
- `7-9重`：重点检验流派完整度，失败成本可控但不可无脑通关。

## 2.3 大境跨越节奏

- `9重 -> 下一境1重` 必须通过双劫挑战（天劫+心魔劫）。
- 跨越后应解锁至少 `1个核心系统入口 + 1个资源新层`。

## 3. 突破流程设计

1. 检查修为是否达到 `RealmBreakConfig.NeedCultivation`。
2. 检查前置挑战（境界试炼/剧情节点）是否完成。
3. 进入渡劫准备（选择策略位、消耗可选增益道具）。
4. 结算雷劫/天劫结果，成功则推进层级。
5. 失败则执行返还、劫灰发放、软保底累计。

## 4. 数值设计要点

## 4.1 需求曲线

```text
小层需求修为 = A(境界) × 层数系数 × 成长倍率
大境跃迁需求 = 当前境9重需求 × 2.5~4.0
```

## 4.2 成长压力控制

- 保证单日可见进度（修为条、材料条、挑战条至少一项可推进）。
- 避免连续两天“零成长反馈”，必要时触发定向补给任务。

## 4.3 失败体验控制

- 不回退大境界，仅回退当前小层进度。
- 返还主要资源80%，并发放劫灰用于失败对冲。

## 5. 代码设计要点（ET）

## 5.1 数据组件

```csharp
public class RealmComponent : Entity, IAwake
{
    public int RealmGroupId;
    public int RealmStage;
    public long CultivationValue;
    public int BreakFailCount;
    public int LastBreakResult;
}
```

## 5.2 逻辑系统

- `RealmComponentSystem`：修为推进、层级推进、解锁事件投递。
- `RealmBreakSystem`：突破前置校验、资源扣除、失败补偿。
- `RealmUnlockSystem`：境界变化后的系统解锁与功能开放。

## 5.3 事件与消息

- 服务端事件：`OnRealmStageChanged`、`OnRealmGroupChanged`。
- 客户端刷新：主界面战力、养成入口、引导任务链。

## 5.4 异步一致性

- 突破流程涉及副本/战斗异步结算时，`await` 后通过 `EntityRef<RealmComponent>` 重取。
- 使用幂等校验避免重复点击导致多次突破结算。

## 6. 配置与协议建议

- 配置：`RealmConfig`、`RealmBreakConfig`、`RealmUnlockConfig`。
- 协议：`C2M_RealmBreakTry`、`M2C_RealmBreakResult`、`M2C_RealmSync`。

## 7. 验收标准

- 新账号7天内可完成多次小层突破与至少一次大境跨越。
- 连续失败用户在48小时内有可见缓解路径。
- 纯PVE玩家境界进度稳定达到SLG玩家的80%-90%。
