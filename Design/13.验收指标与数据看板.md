# 13. 验收指标与数据看板

## 1. 模块目标

- 建立围绕“纯修仙可长线玩”的统一验收口径。
- 将主观体验转为可量化指标，支撑版本决策。
- 为阶段验收、灰度放量、线上调优提供数据基线。

## 2. 核心验收指标

## 2.1 留存与活跃

- 纯修仙玩家30日留存达到立项阈值。
- 纯修仙玩家日活时长稳定在目标区间。
- 纯修仙玩家14日/30日周目标完成率达到版本阈值。

## 2.2 进度公平

- 未参与SLG玩家境界进度不低于SLG玩家的80%-90%。
- 关键主线节点（突破/章节）完成率不因SLG参与出现断层。

## 2.3 体验节奏

- 7日内至少经历10次小层雷劫。
- 14日内至少完成2次大境跨越。
- 30日内仍有明确中期目标（境界/法宝/天梯/图鉴）。

## 3. 负反馈监控指标

- 连续卡关 >72小时用户占比。
- 雷劫失败后次日流失率。
- 因SLG压力导致退坑反馈占比。

## 4. 分层看板设计

## 4.1 用户分层

- 新手层：1-7日。
- 成长期：8-30日。
- 长线层：31日以上。

## 4.2 境界分层

- 凡道、神道、仙道三段分开统计突破成功率与卡关率。

## 4.3 行为分层

- 纯修仙用户。
- 轻度SLG参与用户。
- 深度SLG参与用户。

## 4.4 口径定义

- 纯修仙用户：近7日无宗门活跃或 `slg_time_share <= 5%`。
- 轻度SLG参与用户：近7日有宗门行为，且 `slg_time_share <= 20%`。
- 深度SLG参与用户：近7日 `slg_time_share > 20%` 或持续参与周结算竞争。

## 5. 数据看板字段建议

- `daily_break_count`
- `tribulation_fail_streak`
- `offline_collect_rate`
- `pve_core_resource_gain`
- `slg_reward_ratio`
- `anti_block_trigger_count`
- `solo_week_goal_complete`
- `slg_time_share`
- `solo_midterm_target_depth`

## 6. 预警阈值建议

- 连续失败高于阈值触发版本内热修排查。
- SLG资源占比超过10%触发奖励边界巡检。
- 中期目标完成率跌破阈值触发活动与引导补强。
- `slg_time_share_p90` 超过25%触发减负与入口曝光复核。

## 7. ET落地要点

## 7.1 建议埋点组件

```csharp
public class PlayerProgressStatComponent : Entity, IAwake
{
    public int DailyBreakCount;
    public int TribulationFailStreak;
    public int MainlineChapterProgress;
    public int SlgParticipationLevel;
    public int DailySlgMinutes;
    public int WeeklySoloGoalComplete;
}
```

## 7.2 建议统计系统

- `ProgressStatCollectSystem`：行为埋点采集。
- `ProgressStatAggregateSystem`：日/周指标聚合。
- `ProgressAlertSystem`：阈值告警与自动报告。

## 8. 验收流程

1. 功能验收：链路可跑通。
2. 数值验收：关键曲线在目标区间。
3. 体验验收：卡关与挫败指标可控。
4. 公平验收：纯修仙与SLG玩家进度差合理。
5. 解耦验收：纯修仙用户在不触碰SLG时仍存在清晰的周目标与中期追求。
